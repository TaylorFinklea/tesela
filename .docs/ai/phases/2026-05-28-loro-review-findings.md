## [1] HIGH — BlockDelete does not reparent surviving children — shadow keeps deleted block's children at their original deeper indent while disk reparents them to root
**File:** `crates/tesela-sync/src/engine/loro_engine.rs` — apply_payload_inner, BlockDelete arm, lines 1064-1078 (tree.delete at line 1074)  
**Category:** CRDT block model correctness / engine parity

When a block with children is deleted, the two engines diverge structurally. SqliteEngine::apply_block_delete (sqlite_engine.rs:1000-1010) removes the block AND re-parents every direct child to `parent=None, indent=0`. LoroEngine's BlockDelete arm only calls `tree.delete(node)` on the deleted node and never touches the children's `indent_level` meta. Because the Loro block model is FLAT (children are not tree-children of the deleted node; they are independent root children carrying an `indent_level` meta), deleting the parent node leaves each child at its original deeper indent. I confirmed this empirically: with A (indent 0) and B (child of A, indent 1), after BlockDelete(A) the shadow renders "  - B ..." (indent 1) while SqliteEngine writes "- B ..." (indent 0). dual_engine::structurally_equal compares `a.indent == b.indent`, so this is flagged as a genuine divergence (1 vs 0), not masked by normalization.

**Why it matters:** This is a permanent, self-perpetuating divergence on a common operation (deleting an outline parent that still has children). It blocks getting structural divergence to 0, and it never self-heals because the NoteUpsert reseed path (lines 977-984) only seeds when the tree is empty. A hard cutover making LoroEngine authoritative would make the WRONG indent canonical and write it back to disk, corrupting every outline that had a parent deleted while children survived. The handoff docs (2026-05-26-sync-architecture-redesign.md:118) already flag delete-parent-while-editing-child as a high-risk data-loss class.

**Suggested fix:** In the BlockDelete arm, before tree.delete(node), reset surviving direct children to indent 0 to mirror SqliteEngine. SqliteEngine only reparents DIRECT children (parent == deleted block), so to match exactly the per-node parent must be tracked: store parent_block_id in node meta on BlockUpsert/seed (currently dropped), then on BlockDelete set indent_level=0 for nodes whose parent meta == the deleted block. Add a regression test deleting a parent with a surviving child and asserting the child renders at indent 0.

**Verifier:** Confirmed against the code and reproduced empirically. SqliteEngine::apply_block_delete (crates/tesela-sync/src/engine/sqlite_engine.rs:1005-1010) re-parents direct children of the deleted block to `parent=None, indent=0`. LoroEngine's BlockDelete arm (crates/tesela-sync/src/engine/loro_engine.rs:1064-1078) only calls `tree.delete(node)` + `doc.commit()` and never touches surviving children's indent. The block model is genuinely FLAT: seed_tree_from_flatblocks (lines 700-720) and the BlockUpsert arm (lines 1004-1023) create every block under TreeParentId::Root and store only block_id/text/indent_level meta; parent_block_id is explicitly dropped (`parent_block_id: _`, line 991). A logical child is therefore a Root sibling of its parent, not a tree-child, so tree.delete(parent) removes only that one node and leaves siblings' indent_level meta intact. render_note (lines 418-437) walks tree.children(Root), skips deleted nodes, and reconstructs indent purely from the indent_level meta (flatblock_from_node, lines 606-610), so the surviving child renders at its original deeper indent. dual_engine::structurally_equal (crates/tesela-sync/src/engine/dual_engine.rs:413) compares a.indent == b.indent, so indent 1 (shadow) vs 0 (disk) is a flagged divergence, not masked by normalization. I added a temporary test (A indent 0, B child of A indent 1, then BlockDelete(A)) and the shadow rendered "  - B ..." (indent 1) while SqliteEngine writes indent 0; test removed afterward, tree clean. Existing coverage misses this: block_delete_removes_from_render (lines 1234-1266) deletes a childless sibling. The NoteUpsert reseed path (lines 977-984) only seeds when the tree is empty, so it never self-heals. I lowered severity from the reported `critical` to `high`: it is a real, permanent, self-perpetuating structural divergence that blocks divergence-0 and would write the wrong indent on cutover, but it is reachable only on the specific delete-parent-with-surviving-children case (not all deletes), the engine is still a non-authoritative shadow today, and it corrupts outline structure/indent rather than losing block content (the child text survives), so it is not the highest data-loss tier.

## [2] HIGH — NoteUpsert reseed guard makes any prior shadow/disk block drift permanent — a full-content NoteUpsert never re-syncs an already-populated tree
**File:** `crates/tesela-sync/src/engine/loro_engine.rs` — apply_payload_inner, NoteUpsert arm, lines 977-984 (already_has_blocks guard); contrast sqlite_engine.rs materialize NoteUpsert, line 641 (full-file overwrite)  
**Category:** CRDT block model correctness / convergence

On NoteUpsert, SqliteEngine::materialize does an unconditional full-file overwrite with the op's content (sqlite_engine.rs:641), so the on-disk block set always becomes exactly what the NoteUpsert body parses to. LoroEngine only seeds the tree from the parsed body when already_has_blocks is false; for any note whose shadow tree is already non-empty it updates root meta + page_props but leaves the existing tree blocks untouched. This is correct for the common no-op re-upsert case (avoids duplicate nodes), but it means the shadow tree and the disk file can permanently disagree whenever a NoteUpsert carries a block set that differs from the shadow's current tree. Concrete trigger: the record_sync_update fallback (notes.rs:1176-1198) emits a whole-content NoteUpsert when the block diff is empty but content changed. If the shadow has already drifted from disk (e.g. via the BlockDelete-children bug above, or any best-effort shadow op that failed and was swallowed in dual_engine.rs:529-534), the authoritative full-content NoteUpsert SqliteEngine uses to re-establish truth on disk does NOT re-establish it in the shadow.

**Why it matters:** It removes the one self-healing mechanism you'd expect to converge divergences — a full-content rewrite. Because reseed is suppressed for non-empty trees and shadow op failures are silently dropped, drift accumulates monotonically and only a snapshot wipe + full replay clears it. During the soak this shows up as divergences that never resolve even after the user re-saves the whole note; at cutover the stale shadow blocks would be written to disk.

**Suggested fix:** When a NoteUpsert arrives for a note whose tree is already populated AND the parsed body's block set (by block_id + text + indent) differs from the current tree, reconcile the tree to the body instead of skipping: delete tree nodes whose block_id is absent from the new body, update nodes present in both, and create nodes new to the body — preserving document order. Keep the empty-tree fast path. This makes a full-content NoteUpsert authoritative for the shadow exactly as it is for SqliteEngine. Add a test: seed a note, mutate the shadow tree out of band, fire a NoteUpsert with the canonical body, assert render matches the body.

**Verifier:** Verified against the current code; the finding's mechanism is accurate.

CONFIRMED FACTS:
1. loro_engine.rs:977-984 — NoteUpsert only seeds the tree via seed_tree_from_flatblocks when `already_has_blocks` is false. For an already-populated tree it updates root meta (content/slug/title) + page_props (set_page_properties) but never touches the existing `blocks` tree nodes.
2. sqlite_engine.rs:641 — SqliteEngine::materialize does an unconditional `tokio::fs::write(&path, content)` on NoteUpsert, so on-disk blocks become exactly the op body. The two engines therefore handle a full-content NoteUpsert asymmetrically.
3. seed_tree_from_flatblocks (line 700) is called from only two sites: the boot/load path and the NoteUpsert guard at line 983. There is no reconcile/resync/diff-against-tree path anywhere in tesela-sync (grep confirms).
4. rebuild_index_from_docs (lines 250-301) is NOT a tree self-heal: it reads root `content` meta and rebuilds only the INDEX (title/slug/tags/links via index_upsert + parsed.page_properties). It never re-seeds or reconciles the per-note `blocks` tree.
5. The divergence check (dual_engine.rs:273/301/320 compare_note/compare_all/spawn_divergence_check) is read-only — it reports, it does not repair.
6. Shadow op failures are silently swallowed (dual_engine.rs:529-534, only tracing::warn), so a failed shadow BlockUpsert/Delete leaves the tree drifted while the primary/disk is correct — a reachable precondition.
7. The diff producer (diff.rs:68-119) computes ops against `prev_content` (the on-disk file), not the shadow tree, and with emit_deletes:false. Blocks unchanged in a given edit emit no op (line 113), and deletes are never inferred. So ordinary editing only re-converges the specific block the user touches; it cannot heal drift on untouched blocks or stale-extra blocks. The full-content NoteUpsert fallback (notes.rs:1183) — the operation a user intuitively uses to rewrite a whole note — is the natural recovery, and it is exactly the path the guard suppresses.

NET: once a populated shadow tree drifts from disk, nothing converges it back (boot rebuild=index only, divergence check=read-only, block diff=vs disk + no deletes, NoteUpsert=guarded). Drift is effectively permanent until a snapshot wipe + full replay. At hard cutover the stale shadow blocks would be written to disk. The data-safety concern is genuine.

QUALIFICATION (why high, not critical): this is a recovery-GAP, not a drift-SOURCE — it is conditional on prior drift already existing (a swallowed shadow failure or another tree bug must occur first). The reseed guard's stated rationale ("avoid duplicate nodes on no-op re-upsert") is itself correct, and in the all-happy-path case the NoteUpsert fallback carries a block-identical body (it fires only when ops.is_empty()). The finding's phrase that this is "the one self-healing mechanism you'd expect" slightly overstates the design intent (block ops are the intended convergence path), but the underlying defect — full-content rewrite cannot re-establish shadow truth for a populated tree — is real and matters for the stated goal of driving structural divergence to 0 before cutover. The suggested fix (reconcile populated trees to the body by block_id set-diff, keeping the empty-tree fast path) is the right shape.

## [3] HIGH — Deleted note can be resurrected on disk by a late block op (stale slug from append-only oplog)
**File:** `/Users/tfinklea/git/tesela/crates/tesela-sync/src/engine/sqlite_engine.rs` — find_slug_for_note / find_slug_for_note_uncached (lines 729-796); materialize NoteDelete (lines 649-661); apply_block_upsert (lines 867-920)  
**Category:** data-safety / data-loss

NoteDelete's materialize removes the slug_cache entry for the note (line 655) and removes the file, but the oplog is append-only and still holds the prior NoteUpsert. find_slug_for_note_uncached scans the oplog DESC and returns the display_alias of the most recent NoteUpsert for that note_id (lines 762-771); NoteDelete is NOT matched in that scan, so it never overrides the slug. A block op that arrives AFTER the delete (a deferred/late BlockUpsert from a peer, or a BlockMove/BlockDelete that re-resolves the slug) calls find_slug_for_note -> cache miss -> uncached scan -> returns the OLD slug, then apply_block_upsert writes <slug>.md (lines 883, 911-916), recreating the deleted note's file. The slug also gets re-cached at lines 736-742, so the resurrection persists. The disk-scan fallback (lines 773-793) does not help because the oplog hit short-circuits first.

**Why it matters:** A note the user deleted reappears on disk (and then flows back through the indexer / WS / relay to other devices). This is silent, authoritative-side data resurrection on the engine that writes the real files — exactly the data-safety class the dual-write migration is meant to avoid, and it persists past cutover since the logic is engine-agnostic.

**Suggested fix:** When resolving a slug for a block op, treat a NoteDelete newer than the latest NoteUpsert for the same note_id as authoritative absence (scan should track the most-recent op per note_id and return None if it is a NoteDelete). Equivalently, record a tombstone in slug_cache (e.g. Option<String> or a separate deleted-set) on NoteDelete so find_slug_for_note returns None instead of falling through to the oplog scan, and skip materializing block ops for tombstoned notes.

**Verifier:** Verified against the code; the finding is genuine. (2) The oplog is append-only: delete_note (notes.rs:539-549) records only a NoteDelete via record_sync_delete (notes.rs:1222-1230) and never removes the prior NoteUpsert/BlockUpsert rows. NoteDelete's materialize (sqlite_engine.rs:649-677) only removes the file and drops the slug_cache entry (line 655) — no tombstone is recorded (grep confirms no deleted-set anywhere). (3) A late BlockUpsert then resurrects the file: apply_block_upsert (lines 867-920) does find_slug_for_note -> cache miss -> uncached scan -> returns the stale slug, re-caches it (lines 736-742), reads the path with NotFound mapped to an empty String (line 886) rather than bailing, then unconditionally writes <slug>.md (lines 911-916), recreating the deleted note. (4) Reachability is real: apply_changes (lines 279-337) materializes ops in batch-arrival order via the `for op in ops` loop, NOT re-derived from final HLC state, so a delete-vs-edit race or a deferred/late BlockUpsert from a stale-view peer (a scenario explicitly acknowledged at notes.rs:552-554, where stale clients "stomp peer-added blocks they hadn't fetched yet") lands a BlockUpsert after the NoteDelete has materialized. The finding correctly centers the mechanism on apply_block_upsert, so this does not undermine the core bug. ## [4] HIGH — block_content strips key::value lines from BOTH sides, hiding real block-text divergence
**File:** `crates/tesela-sync/src/engine/dual_engine.rs` — block_content (lines 420-436), used by structurally_equal (lines 403-415)  
**Category:** divergence-false-negative

structurally_equal compares each block's text via block_content(), which filters out every line whose trimmed form parses as `key:: value` (key = alphanumeric/_/-). This filter runs on BOTH primary and shadow before comparison. Any block whose folded continuation lines differ ONLY in their `key:: value` content (e.g. primary has `status:: done`, shadow has `status:: todo`, or one side has `priority:: high` and the other does not) will be reported as Match because those lines are deleted from both `Vec<String>` before equality. The same applies to a top-level block text that itself looks like `foo:: bar`. Because FlatBlock.text folds ALL continuation/property sub-lines into a single string (note_tree.rs lines 373-377), block-level property drift is entirely invisible to the structural check.

**Why it matters:** Block-level properties are real note content (Logseq task status, priority, scheduled dates). If the shadow's BlockUpsert text and the disk file's block text disagree only on these `key:: value` continuation lines, the check reports zero divergence and gives false confidence. At hard cutover LoroEngine becomes authoritative and the disk's property values are replaced by the shadow's stale values — silent data corruption of exactly the metadata that drives task workflows.

**Suggested fix:** For the structural block-content comparison, do NOT strip `key:: value` continuation lines. Compare the full folded block text (after bid-marker stripping + trailing-whitespace trim). If granular block-property merge is genuinely deferred, at minimum compare the key::value lines as an order-insensitive set rather than dropping them, so a value change still surfaces as divergence.

**Verifier:** Verified the full chain end-to-end; the finding is accurate.

(1) `block_content` (dual_engine.rs:420-436) filters out every line whose trimmed form parses as `key:: value` (key = alphanumeric/_/-), via the `match t.find("::")` branch at lines 425-432.

(2) This runs on BOTH sides inside the real divergence decision: `structurally_equal` (line 413) does `block_content(&a.text) == block_content(&b.text)`, and `structurally_equal` is what `compare_note` uses to decide Match vs Diverge (line 283) — not merely the display `normalize`. So any block whose folded continuation lines differ only in `key:: value` content is reported as Match.

(3) Block-property continuation lines are real, mutable, round-tripped content in BOTH engines, so this masks genuine value drift rather than a known omission:
- `set_block_property` (notes.rs:1476) + `upsert_block_property_in_note` (notes.rs:1584) write `key:: value` continuation lines into block bodies — these are exactly the Logseq task status/priority/scheduled fields driving recurrence/dependency workflows.
- `parse_body_blocks` (note_tree.rs:373-377) folds continuation lines into `FlatBlock.text`.
- LoroEngine stores that text verbatim with no stripping: `seed_tree_from_flatblocks` (loro_engine.rs:714) and the BlockUpsert path (loro_engine.rs:1018) both `meta.insert("text", ...)`; `flatblock_from_node` (loro_engine.rs:611-616) reads it back and `render_note` (line 437) serializes it. SqliteEngine likewise carries the full `text` (sqlite_engine.rs:678-702).

So the shadow can hold `status:: todo` where disk has `status:: done` (or have/lack `priority:: high`) and `structurally_equal` returns true because both lines are deleted from the `Vec<String>` before comparison. The only existing safeguard, the `page_properties` equality at line 406, covers leading page-level properties only — nothing compares block-level continuation properties.

Severity assessment: not active corruption today (shadow is non-authoritative, so a stale shadow value harms nothing live), which is why this is high rather than critical. But the divergence check exists precisely to gate the irreversible hard cutover, and this is a blind spot in that gate over real task-workflow metadata: it can report structural divergence 0 while block-property values genuinely differ, and at cutover LoroEngine becomes authoritative and would overwrite disk property values with the shadow's. The code comments (dual_engine.rs:398-402) acknowledge granular block-prop merge is deferred per decisions.md 2026-05-28 — but deferring the merge does not justify the gate silently passing value disagreements; at minimum the key::value lines should be compared as an order-insensitive set so a value change surfaces as divergence. Real, reachable, and a genuine cutover-safety gap. High.

## [5] HIGH — Non-bullet body content (prose/headings) is invisible to both engines, so the check matches while disk content silently differs
**File:** `crates/tesela-sync/src/engine/loro_engine.rs` — render_note (lines 418-438) + apply_payload_inner NoteUpsert arm (lines 922-987); driven by record_sync_update fallback in notes.rs lines 1176-1198  
**Category:** divergence-false-negative

record_sync_update emits a full-content NoteUpsert fallback whenever the block diff is empty but raw content changed (frontmatter OR non-bullet body content such as headings/prose). SqliteEngine.materialize writes that full content to disk verbatim. LoroEngine only models page_properties + bullet blocks: render_note emits page_props + tree blocks and nothing else, and parse_note discards non-bullet body content entirely (note_tree.rs module docs: 'Non-bullet body content currently does not survive the round trip'). So a note containing markdown headings or prose paragraphs will have that content on disk but represented NOWHERE in the shadow. Both materialize_note_body(disk, frontmatter-stripped) and render_note(shadow) parse through parse_note in structurally_equal, which drops the prose on the primary side too — so the comparison sees equal (empty) bodies and reports Match.

**Why it matters:** The divergence check is supposed to gate the cutover that deletes SqliteEngine. Any note with non-bullet content (headings, free prose, raw segments) reports Match today but loses that content the moment LoroEngine becomes the authoritative renderer. This is the most dangerous class of false negative: it is invisible precisely because the same lossy parser is applied to both sides.

**Suggested fix:** Make structurally_equal compare the FULL parsed body including any raw/non-bullet segments (extend NoteTree/parse_note to retain raw_segments and compare them), OR explicitly classify notes containing non-bullet body content as Diverge until LoroEngine models them. Do not let a lossy parser run symmetrically and call the result parity.

**Verifier:** CONFIRMED REAL. The divergence gate produces a false negative for any note containing non-bullet body content (headings, prose paragraphs).

Mechanism, verified end to end:
1. parse_note drops non-bullet prose. In note_tree.rs parse_body_blocks (lines 345-378), a non-bullet line is only appended when `current` is Some (continuation branch, line 373: `else if let Some(rb) = current.as_mut()`). A heading/prose line that appears with `current = None` (e.g. before the first bullet, or after a blank line that ended the prior block's continuation run at lines 347-351) hits neither branch and is silently discarded. There is NO raw_segments field on NoteTree (struct at lines 44-63 has only frontmatter/page_properties/blocks/stamped_any); a grep across the whole tree finds `raw_segments` ONLY in the stale module-doc comment at lines 33-36, which falsely claims the content is "preserved verbatim as raw_segments." That claim is aspirational/wrong — the code does not preserve it.
2. Disk keeps the prose. sqlite_engine.rs materialize NoteUpsert (line 641) writes the full `content` verbatim, and materialize_note_body (lines 153-172) returns that disk content frontmatter-stripped — prose intact.
3. Shadow never renders the prose. loro_engine.rs render_note (lines 418-438) builds a NoteTree from only the tree blocks + read_page_properties and serializes that; the full `content` it stored on root meta (apply_payload_inner NoteUpsert arm, line 938) is never emitted. So the shadow's rendered surface omits the prose entirely.
4. The comparison cancels the loss. dual_engine.rs compare_note (lines 273-296) calls structurally_equal(&p, &s), and structurally_equal (lines 403-415) runs parse_note on BOTH sides and compares only page_properties + blocks. The disk side's prose is dropped by the same parse_note that the shadow side already lacks, so both reduce to identical {page_properties, blocks} and the check returns Match.

Reachability: record_sync_update in notes.rs (lines 1176-1198) explicitly emits a full-content NoteUpsert fallback when the block diff is empty but raw content changed (prose/heading added) — and prose can also be present on disk from legacy FsNoteStore writes. update_note (line 479) routes through this. So real notes hit this path.

Why high not critical: LoroEngine is still a SHADOW — reads come from SqliteEngine, whose disk files retain the prose, so no data is lost today. But this is precisely the cutover-gating false negative: the check certifies Match for prose-bearing notes, and the moment LoroEngine becomes the authoritative renderer those headings/paragraphs vanish. The danger is that a lossy parser runs symmetrically and is called parity. The finding's mechanism, file/line citations, and severity are correct; its only inaccuracy is quoting an older version of the module doc — the current doc is in fact MORE misleading (it claims preservation that does not exist).

## [6] MEDIUM — rebuild_index_from_docs never prunes stale entries — ghost notes survive a schema-version rebuild
**File:** `crates/tesela-sync/src/engine/loro_engine.rs` — rebuild_index_from_docs (lines 250-301); interacts with with_snapshot_dir boot gate (lines 209-238) and load_snapshots_from_dir (527-584)  
**Category:** correctness/data-integrity

rebuild_index_from_docs iterates `self.inner.docs` and calls `index_upsert` for each LOADED per-note doc, but it never clears the index `notes` map first. It is an upsert-merge, not a rebuild — despite the docstring 'Rebuild every index entry from the loaded per-note docs.' Any index entry whose per-note snapshot failed to load is left untouched. load_snapshots_from_dir deliberately skips `.bin` files that are corrupt or have malformed/non-hex names (lines 556-580), so those notes drop out of `docs` while their stale index entry persists. After a boot rebuild (triggered by an INDEX_SCHEMA_VERSION bump), the index then contains ghost entries: notes listed by index_entries() that have no backing doc, for which render_note() returns None.

**Why it matters:** Post-cutover the index is the spine for the note list / ref resolution / backlinks. A ghost entry produces a phantom note in the note list that can't be opened (render_note -> None) and shows up as ShadowMissing in the divergence check, which is exactly the kind of false signal that blocks reaching divergence 0. It also means a NoteDelete that happened while the snapshot was unreadable could leave the deleted note resurrected in the index across the next schema bump.

**Suggested fix:** Before the upsert loop, clear the index `notes` map (e.g. delete all keys via notes.delete over a snapshot of the current keys, or replace the container) so the rebuild is a true full projection of the loaded docs. If preserving entries for temporarily-unloadable docs is intentional, change the docstring and have the corrupt-snapshot path explicitly decide whether to keep or drop the index entry; don't leave it implicit.

**Verifier:** The core mechanical claim is CONFIRMED. `rebuild_index_from_docs` (loro_engine.rs:250-301) never clears the index `notes` map before its upsert loop. It builds `existing` via `index_entries()` (line 252-257) but uses it only as a title/slug fallback (lines 271-282); the loop (line 260) iterates `self.inner.docs` and calls `index_upsert` for each loaded doc (line 285), never `index_remove` for absent ones. So an index entry whose note_id is not present in the freshly-loaded `docs` survives untouched — an upsert-merge, contradicting the docstring "Rebuild every index entry from the loaded per-note docs." (line 241). The corrupt/missing-snapshot skip path is real: `load_snapshots_from_dir` (527-584) `continue`s past malformed-name (556-562) and unparseable-bytes (574-580) snapshots, so those notes drop out of `docs` while `_index.bin` (loaded separately, 199-208) still carries their entries. With a schema bump triggering the rebuild (gate at 219, 229-237), a ghost index entry results. This is a genuine latent defect and a docstring/behavior mismatch.

HOWEVER, the finding's "why_it_matters" is substantially wrong, which lowers severity from high:

1. FALSE: "shows up as ShadowMissing in the divergence check ... blocks reaching divergence 0." Both the periodic check `compare_all` (dual_engine.rs:301-308) and the report endpoint `get_loro_divergence` (notes.rs:313-322) iterate `note_ids()`/`tracked_note_ids()`, which return `self.inner.docs.read().await.keys()` (loro_engine.rs:399-401; dual_engine.rs:611-613) — the docs map, NOT the index. A ghost entry (present in index, absent from docs) is therefore invisible to divergence and can never produce a ShadowMissing. The finding's primary divergence-blocking argument does not hold.

3. Net: a real correctness gap (no pruning; rebuild is not a true projection; docstring overstates) worth fixing before cutover, but benign today because (a) divergence iterates docs not index so it generates no false signal, (b) index_entries feeds only a debug endpoint pre-cutover, and (c) the trigger requires snapshot corruption plus a schema bump. Medium, not high.

## [7] MEDIUM — Comma-join encoding collision: wiki-link targets containing commas corrupt the index link graph
**File:** `crates/tesela-sync/src/engine/loro_engine.rs` — index_upsert lines 333-334 (entry.insert("links", links.join(","))) and index_entries lines 371-376 (split(',')); link target source crates/tesela-core/src/regex_cache.rs:4-5  
**Category:** correctness/data-integrity
