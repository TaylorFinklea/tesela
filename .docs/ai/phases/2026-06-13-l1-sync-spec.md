# L1 Sync ‚Äî minimum key/pairing + cursor-family migration (HA-first)

Author: Opus (Lead), 2026-06-13. Scoped via an Understand‚ÜíDesign‚ÜíSynthesize workflow (8 agents, tree-verified). Decisions made on the workflow's recommended defaults (Taylor said "dive in").

## The finding (scope inversion)

**L1 is NOT a crypto re-architecture.** The workflow verified against the live tree that the hard parts already shipped:

- **Crypto spine complete + enforced** ‚Äî group key ‚Üí HKDF auth key ‚Üí signed-intent register/verify ‚Üí per-request MAC ‚Üí AEAD seal (XChaCha20-Poly1305). The relay sees only ciphertext; holding the auth key cannot recover the group key. (`crypto/relay_auth.rs`, `transport/relay.rs`.)
- **#195 seq black hole already fixed** ‚Äî `store.rs:181` allocates `MAX(MAX(seq), compaction_seq) + 1`. ‚úÖ confirmed.
- **A5 cursor hardening already landed** ‚Äî `scope_to_identity` (`sync_relay.rs:110`) resets the inbound cursor only on a real `(relay_url, group_id)` change; legacy/matching keeps it. ‚úÖ confirmed + already well-tested.
- **iOS A6 (`.relay` writable) shipped** ‚Äî write paths route `.relay` through the engine seam. ‚úÖ confirmed (S0).

So L1 = **one real plumbing bug** + regression fences + a key-at-rest seam + iOS-inviter/keychain polish. The "cursor-family migration" is the *already-correct* A5 reset, EXERCISED + FENCED ‚Äî there is **no new cursor shape and no stored-cursor data migration** to write.

## Decisions (open questions, resolved on recommended defaults)

1. **Trust model** ‚Äî DEFER Ed25519 device identity (keep `GroupMember.ed25519_pubkey` empty). The bearer group key is sufficient for one trusted user's own devices on a self-hosted zero-knowledge relay; docs put identity at "once Savanne/public CF exists." **‚öÝ Consequence: L1 is NOT safe to point at a public/untrusted relay until identity lands.**
2. **Group-key rotation** ‚Äî DEFER; first post-L1 item. `keys.rs` is structured for it. No rotation = a leaked key/pairing-code is permanent group membership.
3. **Key-at-rest** ‚Äî seam (GKS) ‚úÖ shipped; keychain backend (KS) is a follow-up, gated behind file-fallback.
4. **iOS-as-inviter** (FFI) ‚Äî include as fast-follow (cheap); desktop-as-inviter (PV) proves the spine first.
5. **Multi-user seams** ‚Äî NO new seams; the `PairingCode` v2 `#[serde(default)]` pattern already gives a back-compat path to a v3 signed-device-key field.

## ‚öÝ Operational HARD gates (out-of-code; Taylor must set on the HA relay)

These are NOT enforced by L1 code ‚Äî a misconfigured HA box silently breaks the spine:

1. **`--admin-token` SET** on the relay ‚Äî else hijack-recovery is a 404 and a squatted group is unrecoverable.
2. **`TESELA_RELAY_MAX_BODY=16777216`** (task #140) ‚Äî else a >16 MiB cumulative mosaic 413s the snapshot deposit.
3. **Relay URL = the Tailscale 100.x or WAN host, NEVER the LAN IP** ‚Äî a LAN IP breaks off-network devices + can mis-scope cursors.
4. Relay running **0.2.x with snapshot endpoints**.
5. Tauri-embed relay opt-in (drop `TESELA_DISABLE_RELAY=1`) stays **Phase 2**, guarded by one-relay-participant-per-device_id (never run opted-in embed + opted-in standalone together ‚Üí broadcast-cursor corruption). `TESELA_DISABLE_MDNS=1` stays unconditional.

## Explicitly deferred to CF / untrusted-relay (post-L1)

CF Worker deploy + 1 MiB body-cap + registration-replay durability (CF must pass the SAME conformance suite, deployed AFTER HA work); Ed25519 device identity + per-device revocation + group-key rotation (mandatory once Savanne/public CF); Argon2 passphrase key-wrap (only the seam lands now); relay-brokered WAN rendezvous; web non-extractable WebCrypto key; APNs push; LAN P2P (roadmap step 6); WASM web replica; multi-user ACL.

## Risks

1. **Bearer group key has NO per-device revocation** ‚Äî a lost phone ‚áí rotate the whole group key + re-pair every device, and no rotation flow exists yet. Fine for one user's own devices on HA; a full-group-compromise the moment a 2nd human or public relay enters.
2. **Keychain cutover (KS) = silent-divergence trap** ‚Äî mitigated only by the file-fallback-for-one-release ordering.
3. **Operational gates above are silent if unset.**
