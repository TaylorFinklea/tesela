# Loro Cutover - Phase 5/6 Recon + Plan (2026-05-28)

Source: cutover-recon workflow (4 Explore agents). Banked because it precisely maps the integration points for the live-relay flip + FFI swap + iOS.

## CRITICAL DESIGN RESOLUTION (recon agents disagreed; this is the correct call)

**The relay MUST carry Loro update bytes, NOT Vec<EncodedOp>.** One recon agent suggested keeping the Vec<EncodedOp> wire and having LoroEngine just produce/consume ops. That is WRONG and reintroduces the flashing: replaying high-level ops into each device independently-created Loro doc makes tree.create() mint peer-specific node ids, so the "same" block becomes different CRDT nodes that do not merge. Convergence (proven in Phase 4) requires exchanging Loro UPDATE bytes that merge by construction. Therefore Phase 5 is a genuine relay-protocol change: payload = per-note Loro update bytes; cursor = per-note version vectors (producer tracks per-note last-broadcast VV; consumers import idempotently). Protocol version bump required so downlevel peers reject unknown payloads instead of corrupting.

## Verified facts (corrects stale assumptions)
- iOS DOES run a Rust sync engine via FFI (SyncEngineHandle wrapping SqliteEngine) + uses the relay (tickOutbound/tickInbound). Phase 6 = swap that FFI engine to LoroEngine.
- LoroEngine.record_local IS implemented (one recon agent wrongly said stubbed). LoroEngine.produce_changes_since / produce_local_authored_since ARE stubbed (return empty) - that is the real producer gap for relay sync.
- LoroEngine already has doc_version / export_doc_update / import_doc_update (Phase 4) - the per-doc primitives the relay producer/consumer will use.
- Phase 4 two_engines_converge test already proves engine-level convergence via these primitives. So there is NO further safe non-destructive engine increment before the (destructive) live-relay wiring.

## Integration map (file:line)

### relay-tick
The relay sync path encodes a Vec<EncodedOp> from produce_local_authored_since into postcard bytes at sync_relay.rs:225, wraps it in a SyncEnvelope with device/group/nonce/ciphertext, then aead_seals the ciphertext at relay.rs:216 before sending to the relay server. On poll, relay.rs:305-314 decrypts the ciphertext back to the original Vec<EncodedOp> via aead_open, reconstructs the SyncEnvelope envelope, and passes it to engine.apply_changes(peer, envelope) which decode_op_batch decodes the ciphertext back to Vec<EncodedOp> at loro_engine.rs:1086. The HLC cursor (ntp64 i64) drives produce since/paging at sync_relay.rs:208 and is persisted as outbound_cursor_ntp in relay_state.json. Phase 5/6 must replace the Vec<EncodedOp> postcard payload with per-note Loro update bytes while preserving the cursor model; LoroEngine already has doc_version (oplog_vv encode), export_doc_update (updates since version vector), and import_doc_update (merge bytes) at loro_engine.rs:312-353, but the wire format and produce/apply plumbing remain oplog-based.

**Change here:**
- sync_relay.rs:225 — Currently postcard::to_allocvec(&batch.ops) wraps Vec<EncodedOp>; Phase 5 must encode per-note Loro updates instead. The batch structure itself may need per-note version vectors as cursors alongside the single HLC cursor.
- sync_relay.rs:208-210 — Currently PeerCursor::At(HlcTimestamp) derived from oplog; Phase 5 must track per-note version vectors OR maintain HLC as a coarse ordering but also emit per-note version vectors in the envelope metadata/header.
- wire/envelope.rs:24 — ciphertext field currently carries postcard Vec<EncodedOp>; Phase 5 must carry encoded Loro updates. SyncEnvelope structure may gain a metadata field for per-note version vectors or heartbeat.
- loro_engine.rs:1081-1093 apply_changes — Currently decodes Vec<EncodedOp> via decode_op_batch, iterates, calls apply_payload per-op. Phase 5 must decode per-note Loro update bytes, call import_doc_update per-note, aggregate AppliedChanges from per-note results.
- loro_engine.rs:1107-1116 produce_local_authored_since — Currently stubbed (returns empty ProducedBatch). Phase 5 must iterate tracked note_ids, call export_doc_update(note_id, since=version_vector_from_cursor), pack per-note updates into envelope (with per-note version vector headers/metadata).

**Risks:**
- Version vector cursor model differs from HLC: export/import work on Loro VersionVectors (per-actor lamport clocks) not HLC (wall-clock + actor). Phase 5 must decide: (1) embed version vectors in envelope metadata + track per-note cursors in relay_state.json, (2) use HLC as coarse ordering but export_doc_update from last-known-vv per peer (stored where?), or (3) synthetic oplog-like wrapper that re-emits Loro updates as EncodedOps (defeats Loro CRDT semantics). Risk: cursor desync if version vectors aren't persisted/recovered on restart.
- Cursoring granularity: HLC is device-wide; version vectors are per-actor (Loro's actor IDs). A peer may have seen updates from device A's LoroEngine at actor ID X but not X+1. Phase 5 must map device IDs → actor IDs and track per (device, note) version vectors, not per-device HLC. Relay state persistence may explode if stored as JSON.
- Envelope format stability: SyncEnvelope::ciphertext becomes Loro binary instead of postcard ops. Downlevel peers (old phones, iOS running old FFI) won't decode it. Phase 6 must have a protocol bump or TLV wrapper so old code rejects unknown formats gracefully, not silently corrupts.
- AppliedChanges aggregation: produce_local_authored_since must emit multiple ops/updates per tick. Current batch is one atomic unit; Loro may have multi-note changes in one update. Cursor tracking must advance per-note OR per-update (not per-op), risking off-by-one if the batching logic slips.
- DualEngine during cutover: Once LoroEngine becomes primary (Phase 5), DualEngine's primary->shadow fan-out breaks because SqliteEngine reads disk, not oplog. If you keep DualEngine for divergence check, you must re-implement SqliteEngine's produce to use LoroEngine's tracked notes + export deltas, which is expensive. Risk: early retirement of DualEngine without coverage.
- Missing LoroEngine.record_local impl: Currently stubbed to Ok(()). Phase 5 must implement it to call apply_payload for local edits, ensuring materialization parity with apply_changes inbound path. If skipped, local writes won't show up in produce batches.
- Snapshot persistence on iOS: LoroEngine saves snapshots to .tesela/loro/*.bin on apply. iOS FFI (Phase 6) must either (1) rehydrate from on-disk snapshots at startup, (2) pre-populate from oplog on first boot, or (3) use Loro's built-in persistence layer (if LoroEngine::with_snapshot_dir is iOS-compatible). Risk: iOS app boots with empty LoroEngine shadow, causing divergence on first note render if iOS FFI calls render_note before sync catch-up.

**Risks:**
- TWIN-WRITE RACE: Both SqliteEngine.materialize and (new) LoroEngine.materialize could fire during dual-write if not properly gated. NoteUpsert in apply_changes calls SqliteEngine.materialize (line 317), then DualEngine fans to LoroEngine.apply_payload. If LoroEngine also writes, two writers to same {slug}.md in quick succession. File-watcher sees mtime change twice, fires NoteEvent twice, indexer reindex fires twice — expensive and confusing.
- SLUG RESOLUTION DIFF: SqliteEngine.find_slug_for_note scans oplog (O(oplog)) with caching. LoroEngine has slugs in index doc (O(1)) but no cross-note slug→note lookup. If Phase 5 flips read path and a BlockUpsert arrives before its NoteUpsert, LoroEngine may not know the owning note's slug, failing materialization. SqliteEngine defers with a debug log; LoroEngine must do the same or lose the block.
- FILE WATCHER INVERSION: Indexer.initial_index() scans {mosaic}/notes/ on boot. During dual-write, LoroEngine snapshots are in {mosaic}/.tesela/loro/, not {mosaic}/notes/. File-watcher sees only SqliteEngine-written files. When Phase 5 flips to LoroEngine-only, indexer must switch to reading LoroEngine state (via index_entries()) instead of disk scan, OR LoroEngine must also write .md files. Current code path: FsNoteStore.list() → list {mosaic}/notes/. Phase 5: must call engine.index_entries() instead.
- CONCURRENT EDITS ASYMMETRY: SqliteEngine applies block ops to disk file in-place (read, parse, modify, write). Concurrent peer BlockUpserts to the same note mean two readers of the same file—SQLite oplog is the serialization point. LoroEngine has CRDT merge + tree update, so concurrent ops converge in the doc. But when LoroEngine materializes to disk, it's writing a snapshot of merged state, not reading-then-modifying—different failure modes if mid-write file-watcher tries to read.
- INDEX DIVERGENCE: LoroEngine's index doc carries note_id→{title,slug,tags,links}. If index rebuild (rebuild_index_from_docs, line 402) and oplog replay (prepopulate_shadow_from_oplog, line 220) disagree on a note's slug (e.g., stale snapshot + new NoteUpsert in oplog), Phase 5 cutover may orphan notes or duplicate entries. Current safeguard: divergence check (compare_all). Must be passing *zero divergences* for N days before cutover.
- BLOCK ID STAMPING ORDER: record_sync_update (line 1155) calls stamp_block_ids before sync_engine.record_local. SqliteEngine reads the stamped content from the file it just wrote. LoroEngine reads content from the OpPayload passed in, which already has bids stamped. If render_note includes un-stamped comments, file-watcher indexer sees unstamped blocks. Mitigation: render_note must use the same serialize_note path SqliteEngine does.
- SNAPSHOT CORRUPTION RECOVERY: If {mosaic}/.tesela/loro/ snapshot is corrupt/missing, LoroEngine falls back to oplog replay (prepopulate_shadow_from_oplog). If replay succeeds but a file {mosaic}/notes/{slug}.md existed on disk BEFORE the corrupt state, the engine may materialize a different version than what's on disk, and divergence check fires. Current safety: Phase 4 dual-write with divergence check. Phase 5: must ensure all notes are in the shadow before switching authority, else orphaned-disk notes never materialize.
- MATERIALIZATION LATENCY: SqliteEngine.apply_changes materializes inline (synchronous I/O in the apply path). If LoroEngine queues disk writes (e.g., in a task), apply_changes returns before file hits disk. sync_daemon_loop ticks again in 5s, file-watcher may not have fired yet, old state lingers. Phase 5 must keep materialization synchronous or add a flush point before returning from apply_payload.

**Key Risks & Gotchas:**

1. **Loro snapshot persistence on iOS is not yet built** — Phase 6 must add per-note snapshot save/load or iOS will re-replay the full oplog on every cold launch (massive startup latency). Implement BlockMove→Loro translation in a follow-up after Phase 6 stabilizes.

4. **Cross-device version vector sync is not yet tested** — `doc_version()` returns the Loro doc's oplog version vector (encoded VersionVector bytes). `export_doc_update(since: Some(peer_vv))` uses that to export only updates past that vector. The happy path is: peer A calls `doc_version()` → oplog version V_A; peer B calls `export_doc_update(since: V_A)` → incremental updates; peer A calls `import_doc_update(bytes)` to merge. If there's a bug in VersionVector encode/decode or in the delta calculation, one peer can end up with divergent state. **Test thoroughly**: A unit test suite already exists in `loro_engine.rs` (test_import_export_roundtrip line 2065–2090); verify it covers (1) bootstrap (since=None), (2) incremental after apply, (3) out-of-order imports, (4) duplicate imports. The divergence check task on the Mac should catch divergence in the wild, so Phase 5 is an early-warning system before Phase 6 rolls out.

5. **Pairing flow and group identity don't change, but must be tested with LoroEngine** — iOS calls `fetchPairingCode()` to get the Mac's group identity and relay URL (MockMosaicService.swift:604). The pairing code encodes the group_id + group_key; when iOS builds the RelayClientHandle, it uses these to register. Phase 5/6 don't change this, but the integration test (iOS ↔ Mac over the relay) must verify the pair flow still works when Mac is dual-writing (Phase 5) and after the cutover (Phase 6).

6. **Relay delivery order and op-apply idempotency** — The relay broadcasts envelopes in order (seq incrementing). If an envelope arrives out-of-order or a network retry delivers a duplicate, the engine's content-hash dedup catches it (two NoteUpsert ops with the same content hash are idempotent). Loro docs have a different merge guarantee: Loro merges are commutative and idempotent at the doc level (unlike the oplog which is ordered). However, the relay still enforces ordering at the envelope level — iOS's `tickInbound()` polls `relay.poll(since)` which returns envelopes in order from seq+1 onward. When iOS imports the envelope's `ciphertext` (which will contain Loro update bytes in Phase 6), the import order still matters for replay-friendliness (re-reading a doc from snapshot + partial oplog may produce different results if imports are replayed out of order). **Verdict**: The relay's per-device ordering (seq) ensures iOS sees updates in the order the Mac published them. Loro's internal commutativity means imports are safe even if the oplog is replayed, but the envelope ordering is still the source of truth.

## Honest scope assessment (2026-05-28)
