# Relay-as-Spine Multi-Device Sync — Implementation Spec

**Goal:** Route desktop + iOS convergence through the always-on HA-hosted encrypted relay (the spine) so devices converge with the Mac off. NOT a Mac-hub. The relay transport is fully built and tested but dormant; this lights it up, smallest-testable-first.

## Context & constraints

- **Relay-as-spine, Mac-off:** The HA relay (`ghcr.io/taylorfinklea/tesela-relay:0.2.0`, port 8484, TLS/Cloudflare-fronted for WAN) is the durable convergence point. Each device's `tesela-server` (or iOS `RelayTicker`) is a relay *participant*; no device is a hub.
- **The dormant tick is correct, not stubbed.** `sync_relay::tick` (register → bootstrap-from-snapshots → poll/apply/ack + produce/put → time-gated snapshot deposit) and the WS fan-out wiring (`ws_tx` + `ws_delta_tx`) are implemented and integration-tested (`sync_relay.rs:545`). Two gates hold it dormant: (1) absent `[sync.relay]` block in the mosaic `config.toml`; (2) `TESELA_DISABLE_RELAY` env. Lighting it up = clearing those gates per node, then activating it where it's never run (iOS FFI).
- **Single-writer flock:** `acquire_mosaic_lock()` (`tesela-server/src/main.rs:83`) holds `flock(LOCK_EX|LOCK_NB)` on `<mosaic>/.tesela/server.lock`. Only one `tesela-server` owns a mosaic at a time → only one relay participant per `device_id`. The desktop-embed deliberately sets `TESELA_DISABLE_RELAY=1` (`src-tauri/src/main.rs:162`) to avoid being a second writer under the shared `device_id`. **The embed must not become a hub — it stays a loopback Loro-replica; it just gains relay participation when it is the one writer.**
- **Relay URL config:** Sole server-side source is `load_relay_url_from_config()` reading `[sync.relay] url` from `<mosaic>/.tesela/config.toml`. No `TESELA_RELAY_URL` env exists today. `PUT /sync/relay/config` writes the block and returns `restart_required: true` (handle is boot-time only; no hot reload). URL must be reachable from every device → use the Tailscale 100.x IP or the Cloudflare WAN host, never the LAN IP.
- **Auth/crypto:** `RelayClient::new` HKDF-derives `auth_key` from the group key salted with `group_id`. Group key material comes from `.tesela/group_key.bin` + `.tesela/group_id.hex` (existing). No new key plumbing — the relay is zero-knowledge; snapshots seal under a group-only AAD (`snapshot_aad`), envelopes under per-device AAD.

## Phased plan (smallest-testable-first)

### Phase 1 — Activate the spine tick on a plain `tesela-server` (Mac-only, no device)

The whole path already runs *if* both gates clear; prove it end-to-end against the live relay using two local standalone nodes (two mosaics, two `device_id`s) or one standalone node + the Mac's view, before touching the Tauri embed or iOS.

- **Files:** `crates/tesela-server/src/main.rs` (`load_relay_url_from_config` 813, `bring_up_relay_if_configured` 824, tick spawn 900–951), `crates/tesela-server/src/routes/relay.rs` (`put_config`/`get_config`/`status`), `crates/tesela-server/src/sync_relay.rs` (tick, deposit, bootstrap), `crates/tesela-core/src/config.rs` (`RelayConfig`).
- **Change:** No code change required to *activate* — write `[sync.relay] url = "<tailscale-or-WAN>:8484"` into the test mosaic's `config.toml` via the existing `PUT /sync/relay/config` route (mirror that route; do not hand-roll TOML). Confirm `TESELA_DISABLE_RELAY` is unset in the launching shell. If a gap surfaces, the only candidate code change is a `TESELA_RELAY_URL` env fallback at the top of `load_relay_url_from_config` (mirror the existing `TESELA_DEFAULT_MOSAIC`/`TESELA_SERVER_BIND` env-injection pattern in `spawn_server`) — defer unless Phase 2 needs it.
- **Acceptance:** A standalone `tesela-server` with `[sync.relay]` set, on restart: registers (`register_or_recover` + `verify_registration` succeed, `relay_state.json` gets `registered_at`), runs `bootstrap_from_snapshots`, spawns the tick. An edit on node A reaches node B through the relay (Mac otherwise idle) within ~1 poll interval. `GET /sync/relay/status` shows `configured: true`, advancing `inbound_cursor`, null `last_error`. After ≥5 min (or low `TESELA_RELAY_SNAPSHOT_INTERVAL_SECS`), node A deposits a snapshot and the relay GCs ops ≤ watermark; a freshly-started node C bootstraps from snapshots past the GC window.
- **Verify (command):** `cargo test -p tesela-server sync_relay` (existing integration test at `sync_relay.rs:545` must stay green) **and** a named human/script check: with the relay reachable, start two standalone servers on two mosaics, edit in one, `curl <nodeB>/g/sync/relay/status` shows `inbound_cursor > 0` and the note converges. **Mark: needs the live relay reachable (Tailscale/WAN); no iOS device.**

### Phase 2 — Desktop embed participates via the relay (Mac-only)

Make the Tauri-embedded `tesela-server` a relay participant pointed at the HA relay — without making it a hub and without a second writer.

- **Files:** `src-tauri/src/main.rs` (`spawn_server` env block 152–179, `resolve_remote_url`/`desktop.toml` reader 102–129), `crates/tesela-server/src/main.rs` (`bring_up_relay_if_configured`, `load_relay_url_from_config`).
- **Change:** Make the `TESELA_DISABLE_RELAY=1` injection conditional. Mirror the existing `resolve_remote_url()` pattern exactly: add a sibling `resolve_embed_relay_url()` reading a new `relay_url` key from `~/Library/Application Support/tesela/desktop.toml` (same hand-rolled key=value scanner) and/or a `TESELA_EMBED_RELAY_URL` env the shell *reads* (not passes). When it returns `Some`: omit `TESELA_DISABLE_RELAY` from `spawn_server`, and surface the URL to the server — preferably by relying on the mosaic's own `[sync.relay]` block (already shared/iOS-compatible), or by injecting a server-side `TESELA_RELAY_URL` with a matching fallback in `load_relay_url_from_config` (decide source-of-truth: see Open Questions). **Keep `TESELA_DISABLE_MDNS=1` set regardless** — the embed binds loopback and must never mDNS-advertise. `TESELA_DISABLE_PEER_SYNC` is a no-op (LAN data-plane retired); leave it.
- **Acceptance:** With the embed opted in and pointed at the HA relay, the desktop app (no standalone server running) registers + ticks against the relay, converges with another relay participant while the *other* device is the only peer, and `GET /g/sync/relay/status` (through the loopback) reports active. With relay *not* opted in, the embed behaves exactly as today (`TESELA_DISABLE_RELAY` still injected). No second-writer: only one `tesela-server` (the embed) holds the flock; the embed never advertises mDNS.
- **Verify:** Named human check — launch the desktop app opted-in, edit a note on a second relay participant (CLI node or sim), confirm the edit appears in the desktop UI via relay (kill/idle the second node's local HTTP to prove it came through the relay, not direct HTTP). Plus `cargo build -p tesela-server && cargo build` (Tauri) clean. **Mark: Mac-only; needs the live relay.** Landmine: `PUT /sync/relay/config` returns `restart_required: true` — the embed needs a child restart (or app relaunch) for a runtime config change to take; there is no hot-reload.

### Phase 3 — iOS FFI (`put_snapshots` / `fetch_snapshots`) + Swift RelayTicker snapshot path (needs device/sim)

iOS already does the envelope poll/put tick via `SyncCoordinator`; it lacks the *snapshot-compaction* spine half. Without it, iOS falls behind the relay's compaction watermark after the desktop's first snapshot deposit and cannot bootstrap past the GC window when the Mac is unreachable over HTTP.

- **Files:** `crates/tesela-sync-ffi/src/lib.rs` (`RelayClientHandle` impl 995–; `SyncEngineHandle` impl 292–), `crates/tesela-sync/src/transport/relay.rs` (`put_snapshots` 342, `fetch_snapshots` 422 — the inner methods to wrap), `crates/tesela-server/src/sync_relay.rs` (`deposit_snapshots` 356, `bootstrap_from_snapshots` 384 — the server logic to mirror in Swift), `app/Tesela-iOS/Sources/Data/RelayTicker.swift` (`tickOnce`, `ensureCoordinator`/`buildCoordinator`, `bootstrapNoteIfNeeded` 251), `app/Tesela-iOS/Generated/tesela_sync_ffi.swift` (regenerate), `app/Tesela-iOS/Sources/Views/Settings/SyncSettingsView.swift` (status surface).
- **Change (Rust FFI):** Add `put_snapshots` + `fetch_snapshots` to `RelayClientHandle`, wrapping the *inner* `RelayClient` methods directly so the correct group-only `snapshot_aad` sealing happens in Rust — **do not re-implement AEAD in Swift.** Follow the existing `RelayClientHandle` method + `uniffi::Record` shape (mirror `TickInboundRecord`/`PollProbeRecord`): a `SnapshotEntryFfi { stream_id_hex, payload }` record, `put_snapshots(covers_seq, [SnapshotEntryFfi]) -> u64`, `fetch_snapshots() -> { compaction_seq, [SnapshotEntryFfi] }`. Add the note enumeration the deposit loop needs on `SyncEngineHandle` — resolve the slug-vs-note-id question first (Open Questions): either `tracked_note_slugs()` (reuse slug-based `produce_note_delta(slug, sinceVv: nil)`) or `export_snapshot_by_id_hex(noteIdHex)`. `stream_id` stays opaque bytes (the `note_id`-as-stream_id privacy TODO at `relay.rs:339` is out of scope; coordinate any HMAC-key change across server+iOS simultaneously).
- **Change (Swift):** In `RelayTicker.tickOnce`, mirror the server's deposit cadence (`sync_relay.rs:318–340`): persist `last_snapshot_at` in `UserDefaults`, time-gate (match server 300s or tune for mobile — see Open Questions), enumerate notes via the new FFI, deposit via `putSnapshots(coversSeq: inboundCursorSeq, …)`. In `buildCoordinator`, add a `bootstrapFromSnapshots` step (mirror `sync_relay.rs:384`): `fetchSnapshots()` → import each via `importNoteSnapshot` → advance persisted `inboundCursorSeq` to `compaction_seq` **before** restoring the cursor; keep the subsequent `?since=compaction_seq` poll intact for the tail. This supplements/replaces the Mac-HTTP `bootstrapNoteIfNeeded` workaround (`fetchLoroSnapshot`) which fails exactly when the Mac is off.
- **Acceptance:** iOS can deposit snapshots (relay upserts idempotently, MAX-watermark, no regression) and bootstrap from the relay's compacted set on a cold install / long-offline recovery **with the Mac off** (no `/loro/notes/<slug>/snapshot` HTTP dependency). A note edited on the desktop while iOS was offline+past-GC converges on iOS after foreground via `fetchSnapshots` + tail poll. iOS no longer stalls behind the desktop's compaction watermark.
- **Verify:** `cargo build -p tesela-sync-ffi` + regenerate UniFFI bindings + `xcodebuild` clean (trust `xcodebuild`, not SourceKit phantom errors). Named device/sim check (drive a sim): with the Mac's HTTP unreachable but the relay reachable, cold-launch iOS, confirm convergence of a desktop-made edit that's past the GC window. **Mark: needs a device/sim.** BGTask background-tick is **out of scope** for this phase (foreground-only tick is sufficient for the daily-driver case; revisit only if foreground proves inadequate — see Open Questions).

## Risks & landmines

- **Second-writer corruption (flock is necessary but not sufficient):** The flock stops two `tesela-server` *processes* on one mosaic, but the embed and a standalone server are the same binary on the same mosaic with the same `device_id`. Opening the relay on the embed **while a standalone server also has relay enabled** = two relay participants under one `device_id` → broadcast-cursor/HLC corruption. Rule during the transition: exactly one local relay participant per `device_id` at a time. The conditional-disable in Phase 2 must not be the only guard — never run an opted-in embed *and* an opted-in standalone server simultaneously.
- **The embed must stay a loopback replica, not a hub:** Reconcile the Phase 2 change against `src-tauri/src/main.rs:158`'s explicit second-writer warning. `TESELA_DISABLE_MDNS` stays set unconditionally; do not bundle the mDNS decision with the relay decision. Remote-mode (external hub via `resolve_remote_url`) already spawns no child — the embed-relay path applies only to embedded mode.
- **FFI / OLD-FFI device-migration hazard:** Per `project_block_text_crdt` — install the new FFI build on a device *before* doing cross-device edits, or an old-FFI device can produce divergent state. Regenerate `tesela_sync_ffi.swift`; stale bindings = silent breakage.
- **`snapshot_aad` ≠ `envelope_aad`:** The FFI must call the inner `RelayClient::put_snapshots` (group-only AAD) — re-sealing in Swift with the wrong AAD makes receivers fail to open. Same for `fetch_snapshots` decode.
- **Relay auth/URL reachability:** URL in `config.toml` must be the Tailscale 100.x IP or the Cloudflare WAN host, never the LAN IP — off-network devices can't reach a LAN address. Group key material (`group_key.bin` + `group_id.hex`) must already exist; relay registration fails without it.
- **`restart_required: true`:** No runtime relay hot-reload anywhere. Config writes take effect at next boot/child-restart only. Don't assume a `PUT /sync/relay/config` from the web UI activates the running embed.
- **Snapshot-deposit `inbound_cursor > 0` gate:** The sole/first writer never deposits until a second device has written and it has polled ≥1 of those ops. A single-device group won't compact — expected, not a bug.
- **Stale-snapshot lag:** `bootstrap_from_snapshots` jumps `inbound_cursor = compaction_seq` and skips ops ≤ that. Keep the post-bootstrap `?since=` poll to collect the tail; bootstrap is not retried on failure (only a `warn`) — a device with a failing AEAD-open skips bootstrap and re-fetches from cursor 0, which is unavailable if those ops were GC'd.
- **Relay deployment preconditions:** Confirm `ghcr.io/taylorfinklea/tesela-relay:0.2.0` is published + running with snapshot endpoints (`PUT /groups/{id}/snapshot`, `GET /groups/{id}/snapshots`) and `TESELA_RELAY_MAX_BODY=16777216` (task #140) before Phase 1 acceptance — a >16 MiB cumulative mosaic 413s the deposit otherwise.

## Open questions (need Taylor's decision)

1. **Embed relay URL source of truth (Phase 2):** mosaic `[sync.relay]` block (one source, shared with iOS/standalone, works with the existing settings UI) vs. a desktop-only `relay_url` key in `desktop.toml` (decouples the embed from writing relay config into the shared mosaic)? Affects whether `PUT /sync/relay/config` from the web UI is the activation path.
2. **Embed restart mechanism (Phase 2):** Should the Tauri shell gain a restart-child mechanism so `restart_required: true` is honored without a full app quit-relaunch? Today the child is killed only in `RunEvent::Exit`.
3. **Embedded relay-sync: first-class or transitional?** Is embedded-mode-syncs-via-relay a permanent first-class participant, or a stopgap until users always run a standalone hub + remote mode (`resolve_remote_url`, "the multi-device endgame" per `src-tauri/src/main.rs:7-17`)?
4. **iOS depositor vs. consumer (Phase 3):** Should iOS deposit snapshots (`put_snapshots`) or only consume (`fetch_snapshots`)? Confirm the relay's upsert takes MAX watermark for out-of-order/lower-`covers_seq` iOS deposits without regressing compaction. Consumer-only is simpler and lower battery/data.
5. **iOS note enumeration FFI shape (Phase 3):** `tracked_note_slugs()` (reuse slug-based `produceNoteDelta`) vs. `export_snapshot_by_id_hex` (bypass slug; `tracked_note_ids()` returns blake3 note-id hashes with no reverse slug map in the engine)?
6. **iOS snapshot-deposit interval (Phase 3):** Match server 300s, or deposit only once per foreground session / on background transition (battery + cellular-data budget)?
7. **BGTask necessity (deferred from Phase 3):** For "Mac off but relay available," is a `BGProcessingTask` (fires ~only when plugged in + idle) actually useful, or is the foreground-only poll loop sufficient for the daily-driver case?
8. **iOS pairing-code relay_url:** `encode_pairing_code` (`lib.rs:164`) sets `relay_url: None` (iOS = LAN-only codes). Now that iOS is a full relay peer, should iOS-generated pairing codes carry the relay URL — relevant to the Savanne-collaborator pairing-from-iOS scenario?
