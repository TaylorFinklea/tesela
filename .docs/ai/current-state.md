## Architecture at a Glance

- **Rust workspace** (`crates/`): `tesela-core`, `tesela-cli`, `tesela-tui`, `tesela-mcp`, `tesela-server`, `tesela-plugins`. Stable, well-tested.
- **Web client** (`web/`): **SvelteKit 2 + Svelte 5** (runes) + CodeMirror 6 + `@replit/codemirror-vim` + Tailwind v4 + TanStack Query (@tanstack/svelte-query v6) + Tabler Icons. SSR disabled (`export const ssr = false` in `+layout.ts`).
- TypeScript types generated from Rust via `ts-rs` — run `cargo test -p tesela-core --lib export_bindings`.
- WebSocket client with exponential backoff reconnect, wired to TanStack Query cache invalidation.

**Design quality bar:** Linear × Logseq × Zed — craft, restraint, keyboard-first, dark-mode-first.

**Theme system:** role-token CSS variables keyed by `data-theme` on `<html>` — 26 dark + 4 light themes in `web/src/themes.css` (registry in `web/src/lib/themes/index.ts`). The default is **Prism**, a warm-dark palette derived from the app logo (slate / coral / cream); a **Prism Light** variant ships alongside it. iOS mirrors an 18-theme subset in `Theme.swift`.

- **2026-05-21 — iOS voice: root-caused + fixed (transcribe-on-stop)**: capture mic produced no text + the stop button needed 5–10 taps. Fixes, in order: (1) `StreamingVoiceRecorder` published `meterLevel` ~12×/sec but nothing consumed it — `CaptureBar` re-rendered inside `tabViewBottomAccessory` 12×/sec (the missed stop taps); `meterLevel` removed. (2) `transcribeError` / recorder `.failed`/`.denied` were never rendered — silent failure; `CaptureBar` got a `statusLine`. (3) `LocalTranscriptionEngine`'s Whisper/Parakeet caches made `static` (the `engine` computed property rebuilds per recording). (4) `os.Logger` + `VoiceDiagnostics.shared` (`voiceDiag(_:)`) mirror every phase to the unified log AND the capture-bar status line. **Root cause** (surfaced by #4): the status showed `transcription FAILED: Invalid audio data provided. Must be at least 300ms of 16kHz audio` — FluidAudio's `AsrManager+Transcription.swift` guards `audioSamples.count >= 4800`. `StreamingVoiceRecorder` windowed audio into 5s chunks; the trailing chunk after `stop()` was routinely <300ms → rejected. **Fix:** transcribe-on-stop — the recorder now accumulates the whole utterance and transcribes it in ONE pass on `stop()` (FluidAudio chunks long audio internally via its own `ChunkProcessor`, so hand-rolled windowing was unnecessary *and* harmful); the 5s `chunkTimer` is gone; a `minimumSamples` (4800) guard skips sub-300ms clips. **Second root cause** (the next repro showed `recording too short — 1600 samples` for ~5s of speech): `handleInputBuffer` reused one `AVAudioConverter` across every mic tap buffer but signalled `.endOfStream` after each — `AVAudioConverter` is stateful, `.endOfStream` *permanently* finishes it, so only the first buffer (~1600 samples) ever converted and every later one yielded nothing. Fixed: `.noDataNow` instead (out of input *for now*, not forever). **Third root cause** (next repro: `transcription FAILED: MultiArray shape (2 x 1 x 640)`): `LocalTranscriptionEngine` built the Parakeet TDT decoder state via the bare `TdtDecoderState()`, which hardcodes 2 LSTM layers; FluidAudio's own callers (`SlidingWindowAsrManager`, `ChunkProcessor`) always size it from `manager.decoderLayerCount` (v2/v3 = 2, tdtCtc110m = 1). Fixed to `TdtDecoderState.make(decoderLayers: await manager.decoderLayerCount)`. **Fourth root cause** (next repro: `transcription done — 18 chars` but the text never reached the composer): `recorder.onChunk`/`onError` were callback closures that captured the `CaptureBar` SwiftUI **view struct** — firing seconds later they mutated a stale snapshot whose `@State` no longer connected to the live view (the same reason the old `transcribeError` never displayed). - **2026-05-20 — iOS Parakeet voice hang fixed**: starting the mic with a Parakeet model selected hung the app — the stop button did nothing and no transcript appeared, force-quit required. Two compounding bugs in `StreamingVoiceRecorder`, both masked by whisper-tiny's speed: (1) `stop()` `await`ed `transcribeNextChunk()` before setting `state = .idle`, so for Parakeet (first call loads a CoreML model set — tens of seconds) the recorder stayed `.recording` and the stop button looked dead; (2) `transcribeNextChunk()` had no in-flight guard, so the 5s chunk timer kept firing during that slow first load, each tick starting another `transcribe → loadParakeet` — `cachedParakeet` only populates after the first load finishes, so every concurrent call loaded its *own* hundreds-of-MB Parakeet model set (`AsrManager` is a FluidAudio `actor`, genuinely parallel) → iOS jetsam-killed the app before any chunk landed.
